Instructions

The main goal of this project is to teach you the basics of "Continuous Deployment".

You will have to deploy Docker containers on a remote host, and therefore configure it beforehand.

With the concept of automation and replicability in mind, write a main Python script my_deployer.py to run your operations on the remote host.

Your script consist of a CLI containing multiple subcommands intended to manipulate the deployment of containers on the remote host:

  • config
  • build
  • deploy
  • healthcheck

This script may be able to call other Python scripts you wrote, to implement a form of modularity in your project. The script shall be executed on your machine, and will target a remote server: a local Debian 10 "Buster" VM on your workstation.

Note:
The following steps may or may not have interdependencies, do not hesitate to keep on going with the main subject if you ever happen to be stuck at a specific step.
Restrictions
  • Python 3.7 is mandatory.
  • Docker Swarm, Kubernetes or any other orchestrator are strictly forbidden.
  • Everything made by your script must have a local scope, any third-party interaction (SaaS) is strictly forbidden.
Step 1: Docker installation and configuration

The first thing your script must be able to do is make sure that docker-ce 19.03 is properly installed and configured, with the subcommand config

If the remote host happens to lack a Docker installation, you will have to install it using your script. If the Docker version is out-of-date, upgrade it to version 19.03.

Step 2: Microservices

You must build two microservices:

  • the first one is specified by this subject
  • the second can be configured freely, as long as it remains relevant.
Note:
Your different microservices will be located at the root level of your directory.

The first and mandatory microservice will be named checker

Its directory hierarchy is as follows:

$ ls checker
Dockerfile requirements.txt app.py
Copy

It will have 2 routes:

  • GET - /containers/
    • Return all containers (even stopped ones) if the all query parameter is set to true
    • Returns a list of containers currently running on the host otherwise.
  • GET - /containers/{id}, returns the container's information:
    • container's short hash
    • image name
    • version
    • uptime
    • published ports
    • volumes
      The id parameter represents the container's hash, either in long or short form.

The other microservice's implementation is quite free, make sure it is relevant and that you can explain its purpose (for example: do not implement a "hello world" microservice).

Step 3: Building the services

Your script will ensure the building of the microservices' images on the remote host.
Your script should allow building all or only specific microservices using the build subcommand.

Step 4: Deploy the services

With the deploy subcommand, your script will check if a container is present and/or running on the remote host for each of the specified services, and do the following:

  • If no container is running the microservice, your script will deploy and run a container running your service on the remote host.
  • If a container is already running an outdated version of the image you wish to deploy, your script will:
    • Stop the running container
    • Run the new up-to-date image
    • Cleanup the old container
  • In this situation, if the newly-deployed container fails or does not work, your script will bring the container that you just stopped up again.
Step 5: Healthcheck

Your Dockerfiles will implement Healthcheck to ensure your container's stability.

Your main script will provide a healthcheck subcommand to peek at the running container's state on the remote host. The subcommand will restart the containers if they are unhealthy.
Your script must allow to check either all or only some specified containers.

https://docs.docker.com/engine/reference/builder/#healthcheck

Step 6: Module

Your project must include a requirements.txt file with the exact matching versions of the dependencies to install, and provide a setup.py file to install the script on the host.

Bonus
  • Provide a GUI in addition to your CLI.
  • Configure argument autocompletion for my_deployer.py.
  • In the Docker microservice, use docker-py to obtain the exact information.
  • Provide a way of generating a static documentation of your code and project, with examples.
Warning

This project will also evaluate your ability to write clean and re-usable code. The following are mandatory, keep them in mind for the whole duration of the project.

Your project must:

  • Fit-in with the clean code principles, you can find plenty of tools to ensure the quality of your code online. Some of them were mentioned during the previous week.
  • Be autonomous: Upon calling your script, you must not ask for any intervention from the user. Think about handling everything through command-line arguments.
  • Be autocorrective: Your script must be able to be run twice without generating any warning or error. If there is nothing to do, do nothing.
  • Be thoroughly documented.
  • Generate a clean, readable execution log: every action executed by your script, the state of the action, must be logged.
